# Project Requirements — Red Cat Foundation

**Версия:** v1.0
**Дата:** 2026-06-30
**Статус:** утверждено, исторический baseline

> Этот документ — исходный baseline требований, ограничений и предпосылок, на которых строилась Red Cat Foundation. Он зафиксирован неизменным с момента утверждения. Живые документы, которые могут развиваться дальше независимо от этого baseline — [vision.md](vision.md), [principles.md](principles.md), [roadmap.md](roadmap.md) и документы [governance/](../governance/).

## 1. Назначение

Red Cat Foundation — внутренняя инженерная платформа студии Red Cat. Единый источник истины для проектирования цифровых продуктов: дизайн-токены, архитектура компонентов, документация, процессы. Не дизайн-система отдельного продукта, а платформа, на основе которой стартует каждый новый проект студии.

## 2. Цели Foundation

- Устранить дублирование работы дизайнеров, аналитиков и разработчиков между проектами.
- Снять зависимость от устной/разрозненной экспертизы, перенеся знания в единую документированную систему.
- Сократить время старта нового продукта за счёт готового архитектурного и токен-фундамента вместо построения дизайн-системы с нуля каждый раз.
- Ускорить онбординг новых участников команды.
- Обеспечить консистентность процессов и качества проектирования между проектами разного масштаба — от небольших команд до крупных продуктов.
- Создать платформу, которая остаётся актуальной и масштабируемой на протяжении многих лет, развиваясь вместе со студией.

## 3. Роли и сценарии использования

| Роль | Режим взаимодействия |
|---|---|
| Продуктовые дизайнеры (основная аудитория) | Figma-библиотека: компоненты, токены, шаблоны, UX-стандарты, документация |
| Lead Designer / Design Systems Architect | Развитие архитектуры, ревью, RFC-решения, поддержка системы |
| Разработчики | Документация компонентов, спецификации, токены, состояния, поведение. Кодовые библиотеки — будущий этап |
| Бизнес-аналитики | Стандарты описания процессов, шаблоны требований, сценарии |
| Руководители проектов | Контроль качества, единых процессов, состояния проектов |

## 4. Архитектурные принципы

- **Платформенная независимость**: ядро — токены + архитектура компонентов + документация + процессы, без привязки к фреймворку. Реализации под React/Vue/SwiftUI/Compose и т.д. — поверх ядра, без изменения базовой архитектуры.
- **Мультибрендовость по умолчанию**: Foundation не несёт фирменный стиль ни одного продукта. Бренд = набор токенов/конфигурации. Архитектура компонентов, процессы и инженерные принципы — едины для всех проектов студии.
- **Обратная совместимость**: изменения в Foundation не должны произвольно ломать уже запущенные на её основе проекты. Несовместимые изменения допустимы только как осознанное major-изменение, явно документированное и предсказуемое для команд проектов.
- **Модульность**: токены, компоненты, документация и процессы — независимо изменяемые и расширяемые модули, слабо связанные между собой. Это позволяет развивать и заменять отдельные части системы, не затрагивая остальные.
- **Создание нового проекта на основе Foundation**: новый проект формирует собственную Figma-библиотеку, перенося в неё базовую архитектуру Foundation → настраивает брендовые токены проекта → при необходимости добавляет уникальные компоненты → универсальные улучшения после архитектурного ревью возвращаются в Foundation. Foundation выступает стартовой платформой, а не внешней runtime-зависимостью.
- **Независимость проектов после старта**: обновления Foundation не распространяются автоматически. Каждый проект сам решает, синхронизироваться ли с новой версией (полностью / частично / не синхронизироваться). Любое изменение сопровождается документацией о влиянии на проекты.

## 5. Версионирование

Семантическое версионирование (SemVer) для Foundation в целом:

- **Major** — breaking change: изменение архитектуры компонентов, удаление/переименование токенов, изменение API компонентов в Figma, изменение обязательных процессов, изменение структуры Foundation.
- **Minor** — новые возможности, компоненты, процессы, стандарты без нарушения совместимости.
- **Patch** — исправления ошибок, улучшения документации, оптимизация существующих решений.

## 6. Управление и процесс изменений

- Финальное архитектурное решение принимает Lead Designer / Design Systems Architect.
- Любой участник может предложить улучшение.
- Изменения, влияющие на архитектуру, проходят процесс RFC (обсуждение → принятие / доработка / отклонение).
- Журнал архитектурных решений (ADR) сознательно не вводится на первом этапе — решения фиксируются в соответствующих документах репозитория по мере появления. Вопрос ADR пересматривается, когда система и объём решений вырастут.

## 7. Документация и инструменты

- Язык документации: русский (на первом этапе). Английский — точечно для технических разделов в будущем, не приоритет сейчас.
- Инструменты: только **GitHub** (единый источник документации, знаний, архитектурных решений) и **Figma** (проектирование и сопровождение дизайн-системы). Дополнительные инструменты (Storybook, Zeroheight и т.п.) не вводятся до формирования фундамента.

## 8. Последовательность развития (этапность)

1. **Этап 1 (текущий приоритет)** — технический фундамент: дизайн-токены, архитектура компонентов, структура дизайн-системы, базовая документация.
2. **Этап 2** — UX-стандарты (паттерны навигации, форм, пустых состояний и т.д.) поверх стабильного фундамента.
3. **Этап 3** — DesignOps-процессы команды (как берётся задача, ревью, передача в разработку).

Технический фундамент и регламенты работы команды сознательно не строятся параллельно.

## 9. Принцип объёма на старте

Минимальная достаточная сложность: не проектируется полный скелет всех будущих разделов и не создаются пустые заглушки "на будущее". Каждый новый раздел добавляется только тогда, когда для него сформированы реальные требования. Развитие — эволюционное, по мере использования и накопления опыта.
