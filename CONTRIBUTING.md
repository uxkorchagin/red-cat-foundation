# Contributing

Правила участия в развитии Red Cat Foundation.

## Как предлагать изменения

Полный процесс описан в [governance/contribution-process.md](governance/contribution-process.md). Кратко:

- Небольшие правки (опечатки, формулировки, примеры) — вносятся напрямую, проходят ревью.
- Архитектурные изменения (структура репозитория, принципы, архитектура компонентов, ключевые процессы) — оформляются как RFC по шаблону [templates/rfc.md](templates/rfc.md) и проходят обсуждение перед принятием.

## Процесс RFC

См. [governance/contribution-process.md](governance/contribution-process.md). Финальное решение по архитектурным изменениям принимает Lead Designer / Design Systems Architect.

## Требования к документации

- Терминология должна соответствовать [constitution/glossary.md](constitution/glossary.md).
- Документ должен быть понятен читателю без дополнительного устного контекста.
- Каждое архитектурное решение должно иметь явное обоснование (принцип "Обоснованность", [constitution/principles.md](constitution/principles.md)).

## Требования к качеству изменений

- Изменение не должно дублировать уже существующее решение (принцип "Консистентность").
- Изменение, ломающее совместимость, классифицируется как major и сопровождается описанием влияния на проекты (см. [governance/versioning.md](governance/versioning.md)).
- При недостатке информации для решения — вопрос фиксируется явно, решение не принимается на основе предположений (принцип "Отсутствие предположений").

## Процесс ревью

См. [governance/review-process.md](governance/review-process.md) и соответствующие чек-листы в [checklists/](checklists/).
