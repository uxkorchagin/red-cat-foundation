# Token Architecture

Утверждённая структурная архитектура дизайн-токенов Red Cat Foundation. Это инфраструктурный слой Design Language (см. [README.md](README.md)) — он не содержит перцептивного смысла сам по себе, но определяет, как его выразят все остальные подсистемы: [Color](color.md), [Typography](typography.md), [Spacing](spacing.md), [Radius](radius.md), [Iconography](iconography.md), [Grid](grid.md), [Breakpoints](breakpoints.md), [Motion](motion.md), [Elevation](elevation.md).

## Уровни токенов

Трёхуровневая модель: **Global → Semantic/Alias → Component**.

### Global

Путь: `категория/шаг-шкалы` — например, `color/blue/500`, `space/4`.

Литеральное значение, без смысла использования. Не зависит от режима (см. "Режимы" ниже) — значение шага шкалы одинаково во всех брендах, темах и вьюпортах.

### Semantic / Alias

Путь: `категория/роль/вариант(/состояние)` — например, `color/background/danger`, `color/text/primary/hover`.

Значение — **обязательно ссылка** на Global-токен, никогда не литерал. Это правило — механизм, который делает смену бренда, темы или вьюпорта конфигурацией поверх структуры, а не редактированием самой структуры. Именно на этом уровне применяются режимы.

### Component (опционально)

Путь: `компонент/свойство(/состояние)` — например, `button/background/hover`.

Значение — ссылка на Semantic-токен. Создаётся только тогда, когда компоненту действительно нужно отклонение от прямой семантической привязки — не для каждого токена по умолчанию (принцип минимальной достаточной сложности). Прямая ссылка на Global, минуя Semantic, — антипаттерн, требующий явного обоснования в `knowledge/foundations/`. Component-токен не объявляет собственную вариативность по режимам — она наследуется через ссылку на Semantic-токен.

## Конвенция именования

Слэш-группировка (`category/role/variant/state`), а не точечная нотация — выбрана, потому что нативно соответствует тому, как Figma группирует Variables, а Figma на сегодняшнем этапе — единственный реальный инструмент реализации (см. [constitution/roadmap.md](../../constitution/roadmap.md)). При появлении `platform/implementations/` слэш-путь механически и однозначно транслируется в точечную или kebab-нотацию для кода — структура от этого не меняется.

**Имя токена не включает режим.** `color/background/primary` — одно и то же имя во всех брендах, темах и вьюпортах; меняется только то, на какой Global-токен оно ссылается в активном режиме.

## Режимы (modes)

Режимы — независимые ортогональные оси, не единое перечисление. Каждая ось — отдельная "коллекция" в терминах Figma Variables; токен варьируется только по тем осям, по которым это действительно нужно, а не по всем по умолчанию.

Заведены три оси:

| Ось | Значения | Где определяются точные значения |
|---|---|---|
| **Brand** | По числу брендов студии | Конфигурация конкретного проекта, не сама архитектура |
| **Theme** | Light / Dark | Подсистема [Color](color.md) |
| **Viewport** | Compact / Medium / Expanded (абстрактные классы доступного пространства, не типы устройств — см. [breakpoints.md](breakpoints.md)) | Пороговые значения (px) — Foundation-инвариант, ещё не заполнен, см. `breakpoints.md` |

Режим применяется **только на Semantic/Alias-уровне**. Global-токены — режимонезависимые константы. Component-токены наследуют вариативность через ссылку на Semantic, не объявляют её сами.

Добавление новой оси режима в будущем (например, плотности интерфейса) — minor-изменение: токены, не объявившие зависимость от неё, не затрагиваются. Изменение набора значений уже существующей оси — потенциальный breaking change, классифицируется по [governance/versioning.md](../../governance/versioning.md).

## Статус токена

Active / Deprecated / Removed. Deprecated требует указания замены и периода до удаления в major-версии, согласно [governance/versioning.md](../../governance/versioning.md).

## Правила развития

- Новая потребность бренда или контекста решается на уровне Alias-конфигурации (значение по оси режима), не изменением структуры. Если структуры недостаточно — сигнал расширить её через RFC, а не вводить точечное исключение.
- Изменение самой Token Architecture (уровней, нейминга, набора осей) — всегда breaking change.
- Любое структурное изменение проходит через алгоритм Skill `design-systems-architect`: исследование → минимум две альтернативы → компромиссы → рекомендация → утверждение пользователем.
