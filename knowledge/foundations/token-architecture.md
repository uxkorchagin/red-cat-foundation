# Обоснование: Token Architecture

Зафиксировано Skill `design-systems-architect` после утверждения пользователем. Итоговая архитектура — в [platform/tokens/token-architecture.md](../../platform/tokens/token-architecture.md).

## Исследование

Изучены W3C Design Tokens Community Group (формирующийся стандарт атрибутов токена), CTI-паттерн Style Dictionary (Amazon), трёхуровневая модель Material 3 / Adobe Spectrum / Salesforce Lightning / IBM Carbon, и нативный механизм Figma Variables (независимые коллекции, каждая — со своими режимами, применяемые ортогонально друг другу на уровне фрейма/инстанса).

## Решение 1 — количество уровней

**Альтернатива A — два уровня (Global → Semantic).** Отклонена: единичное исключение для компонента либо засоряет семантический слой, либо требует жёстко прописанного значения в обход системы.

**Альтернатива B (принята) — три уровня (Global → Semantic/Alias → Component)**, с опциональным Component-уровнем. Соответствует масштабу, к которому стремится Foundation, и уже было заложено неявно в `templates/token-spec.md` (поле "Уровень: Global/Alias/Component" существовало до формального решения) — выбор A создал бы противоречие с уже принятым артефактом.

## Решение 2 — конвенция именования

**Альтернатива 1 — точечная нотация** (`color.background.danger.hover`), стандарт W3C и большинства кодовых реализаций.

**Альтернатива 2 (принята) — слэш-группировка** (`color/background/danger/hover`). Figma — единственный реальный инструмент реализации сегодня (кода ещё нет, см. `constitution/roadmap.md`), поэтому имя оптимизировано под ближайшую конкретную потребность. Переход к точечной/kebab-нотации для кода в будущем — механическая, детерминированная трансформация, не требующая пересмотра структуры.

## Решение 3 — режимы (modes)

**Альтернатива A — единый плоский режим**, объединяющий бренд, тему и вьюпорт в одно перечисление (`brand-a-light-desktop`). Отклонена: количество комбинаций растёт перемножением измерений, не масштабируется.

**Альтернатива B (принята) — независимые ортогональные оси режимов**: Brand, Theme (light/dark), Viewport (desktop/tablet/mobile) — отдельные "коллекции", применяемые независимо. Напрямую соответствует нативному механизму Figma Variables и не создаёт комбинаторного взрыва. Режим применяется только на Semantic-уровне; Global остаётся режимонезависимым; Component наследует вариативность через ссылку, не объявляет её сам. Имя токена не включает режим — меняется только то, к какому значению оно резолвится.

## Почему это решение долговечно

Структура отделяет литеральное значение (Global) от смысла использования (Semantic) и от точечного исключения компонента (Component); режимы — отдельное, ортогональное измерение поверх Semantic-уровня. Новый бренд, новая тема или новый вьюпорт заполняют существующие оси значениями и не требуют пересмотра структуры. Изменение самой структуры (уровни, нейминг, набор осей) остаётся редким и явно классифицированным как breaking change.
