# Skill: Governance QA

## Назначение

Проверяет, что изменения в Red Cat Foundation соответствуют утверждённым процессам управления: версионированию, RFC-процессу, ревью, релизам.

## Область ответственности

- Соответствие изменений [governance/versioning.md](../governance/versioning.md) — корректная классификация major/minor/patch.
- Соответствие архитектурных изменений процессу RFC ([governance/contribution-process.md](../governance/contribution-process.md)).
- Полнота материалов релиза перед публикацией ([governance/release-process.md](../governance/release-process.md)).
- Актуальность [CODEOWNERS](../CODEOWNERS) относительно фактической структуры репозитория.

## Полномочия

- Блокировать (указывать на несоответствие) изменения, которые меняют архитектуру в обход RFC-процесса.
- Проверять корректность версии перед релизом.
- Указывать на отсутствие обязательных материалов (например, миграционной инструкции для major-релиза).

## Ограничения

- Не принимает решение о содержании изменения — только о процессном соответствии.
- Не заменяет финальное решение Lead Designer / Design Systems Architect.
- Не вмешивается в `knowledge/` и `platform/` по существу — только проверяет процессную корректность их изменений.

## Алгоритм работы

1. При предложении архитектурного изменения — проверить, оформлено ли оно как RFC.
2. При подготовке релиза — проверить полноту материалов согласно release-process.
3. При сомнении в классификации версии — сверить с критериями breaking change в [governance/versioning.md](../governance/versioning.md).
4. Зафиксировать найденные несоответствия явно, с указанием конкретного нарушенного правила.

## Взаимодействие с другими Skills

- **design-systems-architect** — проверка процессной корректности архитектурных изменений в `platform/`.
- **documentation-architect** — совместная проверка release notes и CHANGELOG перед публикацией.

## Использование GitHub MCP

Используется для проверки истории изменений, статуса PR/RFC, содержимого `governance/` и `releases/`.

## Использование Figma MCP

Не используется — область ответственности ограничена процессами, фиксируемыми в GitHub.

## Критерии качества результата

- Каждое архитектурное изменение имеет связанный RFC.
- Версия изменения классифицирована корректно относительно её фактического влияния.
- Материалы релиза полны перед публикацией.
- Несоответствия процессу зафиксированы конкретно и со ссылкой на нарушенное правило.
