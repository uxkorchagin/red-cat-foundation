# Skills — Framework

Skills — специализированные исполнители, обслуживающие развитие Red Cat Foundation. Этот документ — устав домена `skills/`: он определяет, что такое Skill, как он устроен, как взаимодействует с остальными доменами и как проходит свой жизненный цикл. Конкретные Skills описывают только, что они делают; правила, общие для всех Skills, фиксируются здесь — один раз, а не в каждом файле.

Обязательный шаблон для любого нового Skill — [templates/skill.md](../templates/skill.md).

## 1. Что такое Skill

Skill — это специализированный, постоянно существующий модуль ответственности внутри Foundation: у него есть имя, граница ответственности, полномочия, ограничения, воспроизводимый алгоритм действий, явный контракт взаимодействия с другими Skills и доменами. Skill подчиняется тому же жизненному циклу, что и любое архитектурное решение Foundation: проектирование → ревью → принятие → использование → пересмотр.

## 2. Чем Skill отличается от обычного промпта

| | Промпт | Skill |
|---|---|---|
| Время жизни | Разовый, для одной задачи | Постоянный, переиспользуемый |
| Ответственность | Не определена формально | Явно очерчена и не пересекается с другими Skills |
| Воспроизводимость | Зависит от формулировки в моменте | Алгоритм зафиксирован, поведение предсказуемо |
| Контракт | Отсутствует | Явные вход/выход, явное взаимодействие с другими Skills |
| Управление изменениями | Не версионируется | Подчиняется [governance/versioning.md](../governance/versioning.md) |
| Ревью | Не проходит ревью | Проходит [governance/review-process.md](../governance/review-process.md) и квалификационные критерии (раздел 9) |

## 3. Архитектура Skill

Skill состоит из шести функциональных слоёв:

1. **Идентичность** — имя, статус, версия, владелец, дата последнего пересмотра.
2. **Контракт ответственности** — назначение, что входит и не входит в зону ответственности, полномочия, ограничения.
3. **Интерфейс** — что Skill принимает на входе и что гарантированно производит на выходе.
4. **Алгоритм** — воспроизводимая последовательность действий.
5. **Зависимости** — какие домены Foundation и какие внешние системы Skill использует, и как именно.
6. **Контракт качества** — критерии, по которым результат работы Skill считается приемлемым.

Полный шаблон, реализующий эти шесть слоёв, — [templates/skill.md](../templates/skill.md).

## 4. Как Skills взаимодействуют между собой

Skills не вызывают друг друга напрямую. Foundation использует orchestrator-worker модель: координацию нескольких Skills для задачи, затрагивающей несколько областей, выполняет Claude, руководствуясь разделом "Порядок использования Skills" в [CLAUDE.md](../CLAUDE.md). Раздел Skill «Взаимодействие с другими Skills» описывает не вызов, а контракт передачи — в какой момент результат одного Skill становится входом для другого, и какие Skills потенциально пересекаются по границе ответственности.

Децентрализованная модель (Skills, вызывающие друг друга напрямую) не используется сознательно — она резко увеличивает сложность отладки и непредсказуемость при росте числа Skills. Пересмотр этой модели возможен только через RFC, если масштаб Foundation потребует по-настоящему автономных параллельных агентов.

## 5. Как Skills используют CLAUDE.md

`CLAUDE.md` — оркестратор. Skill не переопределяет границу автономии и иерархию источников истины, зафиксированные там — он действует внутри них. Если полномочия Skill формально входят в противоречие с границей автономии `CLAUDE.md`, побеждает `CLAUDE.md`, и это противоречие — повод пересмотреть Skill через ревью.

## 6. Как Skills используют Constitution

Алгоритм каждого Skill должен быть прослеживаем до принципов [constitution/principles.md](../constitution/principles.md) — так же, как сами принципы прослеживаются до убеждений [constitution/vision.md](../constitution/vision.md). Если задача, попавшая в Skill, требует нарушить принцип, Skill не нарушает его самостоятельно, а эскалирует как RFC.

## 7. Как Skills используют Knowledge

Knowledge — одновременно вход и выход для содержательных Skills. Перед предложением решения Skill обязан свериться с релевантным разделом [knowledge/](../knowledge/), чтобы не дублировать уже зафиксированный опыт. После принятия решения Skill обязан зафиксировать его обоснование в соответствующем разделе Knowledge — прямая реализация принципа "Знание фиксируется, а не подразумевается" ([constitution/principles.md](../constitution/principles.md)).

## 8. Как Skills используют Platform

Для Skills, работающих непосредственно с техническим ядром (например, `design-systems-architect`), [platform/](../platform/) — основной объект работы: они читают и предлагают изменения в спецификациях. Для остальных Skills `platform/` — только источник контекста, который они не модифицируют по существу.

## 9. Жизненный цикл Skill

`Запланирован` (зафиксирован как намерение в [constitution/roadmap.md](../constitution/roadmap.md), файла ещё нет) → `Спроектирован` (черновик по [templates/skill.md](../templates/skill.md)) → `Принят` (прошёл ревью и квалификационные критерии, файл появляется в `skills/`) → `Активен` (используется) → `Пересмотрен` (изменение через RFC при изменении границы ответственности или алгоритма) → `Устарел/Объединён` (ответственность исчезла или перешла другому Skill — файл архивируется, а не удаляется молча).

## 10. Как Skills обновляются

Формальные правки и уточнения формулировок вносятся напрямую, как любая мелкая правка документации. Изменения, затрагивающие область ответственности, полномочия, ограничения или алгоритм, — обязательный RFC, поскольку меняют контракт, на который полагаются `CLAUDE.md` и потенциально другие Skills. Классификация изменения (major/minor/patch) проводится по правилам [governance/versioning.md](../governance/versioning.md): переименование или сужение ответственности Skill — breaking change для всего, что на него ссылается.

## 11. Ревью Skills

Проводится по [governance/review-process.md](../governance/review-process.md), с проверкой квалификационных критериев (раздел 9 ниже) как предметного дополнения к общему процессу — по аналогии с тем, как `checklists/component-review.md` дополняет общий процесс ревью для компонентов.

## 12. Квалификационные критерии перед добавлением в Foundation

- Область ответственности сформулирована однозначно и не пересекается с уже существующими Skills.
- Алгоритм воспроизводим — не зависит от того, кто и в каком настроении его интерпретирует.
- Полномочия и ограничения не противоречат `CLAUDE.md` и `constitution/principles.md`.
- Все обязательные разделы [templates/skill.md](../templates/skill.md) заполнены содержательно — без формальных заглушек.
- Явно описано хотя бы потенциальное пересечение с соседними Skills.
- Ответственность Skill реально сформирована на текущем этапе [constitution/roadmap.md](../constitution/roadmap.md) — не создаётся "на будущее".
- Skill прошёл ревью (раздел 11).
