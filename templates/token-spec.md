# Token Spec: [Название токена / группы токенов]

> Шаблон-заготовка. Не содержит конкретных данных проекта. Структурные правила — см. [platform/tokens/token-architecture.md](../platform/tokens/token-architecture.md).

## Назначение

Что описывает этот токен и какую роль он играет в Design Language.

## Уровень

Global / Semantic (Alias) / Component.

Если Component — обоснование, почему недостаточно прямой ссылки на Semantic-токен (см. правило о точечных исключениях в Token Architecture).

## Путь (имя)

Слэш-нотация согласно уровню:
- Global: `категория/шаг-шкалы`
- Semantic: `категория/роль/вариант(/состояние)`
- Component: `компонент/свойство(/состояние)`

Имя не включает режим — режим влияет только на то, к какому значению путь резолвится.

## Значение

- **Global** — литеральное значение.
- **Semantic / Component** — ссылка на токен уровня ниже. Никогда не литерал.

## Оси вариативности

По каким осям режима варьируется значение этого токена (Brand / Theme / Viewport — все, некоторые, или ни одной). Только для Semantic-уровня; Component наследует вариативность через ссылку.

Если варьируется — указать значение (ссылку) для каждого релевантного режима.

## Платформенная независимость

Как значение должно проецироваться на разные платформы реализации (без жёсткой привязки к конкретной технологии).

## Связь с брендом

Является ли токен брендозависимым (настраивается проектом через ось Brand) или базовым (общим для всех проектов на Foundation).

## Статус

Active / Deprecated / Removed. Если Deprecated — указать замену и версию, в которой токен будет удалён.

## История изменений

Ссылка на RFC, если токен вводился/менялся через RFC-процесс.
